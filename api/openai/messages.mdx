title: "openai package — Message types"
description: "Reference for the Message struct, helper constructors, and the ToolCall types used to build and interpret conversation histories in hrns."
---

Messages are the primary unit of conversation in the OpenAI chat completions API. The `openai` package provides a `Message` struct that covers every standard role, plus helper constructors for the most common cases. Custom or model-specific JSON fields are preserved transparently via the `Extra` map so no information is lost during round-trips.

## Message struct

```go
type Message struct {
    Role       string     `json:"role"`
    Content    any        `json:"content,omitempty"`
    Name       string     `json:"name,omitempty"`
    ToolCalls  []ToolCall `json:"tool_calls,omitempty"`
    ToolCallID string     `json:"tool_call_id,omitempty"`
    Refusal    string     `json:"refusal,omitempty"`

    Extra map[string]any `json:"-"`
}
```

<ResponseField name="Role" type="string" required>
  The speaker role: `"system"`, `"user"`, `"assistant"`, or `"tool"`.
</ResponseField>

<ResponseField name="Content" type="any">
  The message body. Typed as `any` to support both plain strings and structured content arrays (for multimodal inputs such as images). When accumulating streaming deltas, hrns concatenates string content automatically.
</ResponseField>

<ResponseField name="Name" type="string">
  An optional display name for the participant. Passed through to the API as-is.
</ResponseField>

<ResponseField name="ToolCalls" type="[]ToolCall">
  Tool invocations requested by the assistant. Present on `"assistant"` messages when the model calls one or more tools.
</ResponseField>

<ResponseField name="ToolCallID" type="string">
  The ID of the tool call this message is responding to. Required on `"tool"` role messages.
</ResponseField>

<ResponseField name="Refusal" type="string">
  Populated when the model refuses a request. Corresponds to the `refusal` field in the OpenAI response schema.
</ResponseField>

<ResponseField name="Extra" type="map[string]any">
  Captures any JSON fields not listed above. This field is excluded from marshalling by default (`json:"-"`); instead, `MarshalJSON` merges `Extra` keys back into the top-level object, and `UnmarshalJSON` populates it with any unknown keys received from the server.

  This is particularly useful for models that emit non-standard fields such as `reasoning_content` (chain-of-thought reasoning tokens from certain providers).

  ```go
  if reasoning, ok := msg.Extra["reasoning_content"].(string); ok {
      fmt.Println("Model reasoning:", reasoning)
  }
  ```
</ResponseField>

## Helper constructors

### SystemMessage

```go
func SystemMessage(content string) Message
```

Returns a `Message` with `Role: "system"` and the given string as `Content`. Use this for instructions that apply throughout the conversation.

### UserMessage

```go
func UserMessage(content string) Message
```

Returns a `Message` with `Role: "user"`. This is the standard way to represent a human turn.

### ToolMessage

```go
func ToolMessage(content, toolCallID string) Message
```

Returns a `Message` with `Role: "tool"`, `Content` set to the tool's output, and `ToolCallID` set to the matching call ID. This must follow an assistant message that contains a `ToolCalls` entry with the same ID.

## ToolCall and ToolCallFunction

The `ToolCall` type represents a single tool invocation requested by the model. It appears inside `Message.ToolCalls` on assistant messages.

```go
type ToolCall struct {
    ID       string           `json:"id,omitempty"`
    Type     string           `json:"type,omitempty"`
    Function ToolCallFunction `json:"function,omitempty"`
    Extra    map[string]any   `json:"-"`
}

type ToolCallFunction struct {
    Name      string         `json:"name,omitempty"`
    Arguments string         `json:"arguments,omitempty"`
    Extra     map[string]any `json:"-"`
}
```

<ResponseField name="ToolCall.ID" type="string">
  The unique identifier for this tool call, generated by the model. Pass this value as `toolCallID` when constructing the corresponding `ToolMessage`.
</ResponseField>

<ResponseField name="ToolCall.Type" type="string">
  Always `"function"` for function-calling tools in current OpenAI-compatible APIs.
</ResponseField>

<ResponseField name="ToolCall.Function" type="ToolCallFunction">
  The function invocation details.
</ResponseField>

<ResponseField name="ToolCallFunction.Name" type="string">
  The name of the function the model wants to call.
</ResponseField>

<ResponseField name="ToolCallFunction.Arguments" type="string">
  A JSON-encoded string of the arguments the model has chosen. Parse this with `json.Unmarshal` to extract individual parameters.

  ```go
  var args struct {
      Path string `json:"path"`
  }
  if err := json.Unmarshal([]byte(tc.Function.Arguments), &args); err != nil {
      log.Fatal(err)
  }
  ```
</ResponseField>

Both `ToolCall` and `ToolCallFunction` carry their own `Extra` map for the same round-trip preservation reasons as `Message`.

<Note>
  `Content` is typed as `any` deliberately. While most messages use plain strings, the OpenAI API supports structured content arrays for multimodal inputs (e.g. passing an image URL alongside text). Passing a `[]map[string]any` value is valid and will marshal correctly.
</Note>

## Building a conversation

```go
messages := []openai.Message{
    openai.SystemMessage("You are a helpful assistant."),
    openai.UserMessage("What files are in this directory?"),
}

// After the model responds with a tool call:
assistantMsg := // ... response.Choices[0].Message
messages = append(messages, assistantMsg)

// After executing the tool:
messages = append(messages, openai.ToolMessage(result, assistantMsg.ToolCalls[0].ID))
```
